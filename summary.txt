Guitar — Chords & Scales: project summary
==========================================

Goal
----
A client-side web app for exploring guitar chords and scales. Displays note
positions on a fretboard SVG. Supports any tuning, any root note, a wide
range of chords and scales. Primary distribution is a single self-contained
HTML file (no server, no network, no external dependencies).


Architecture
------------
Three concerns are kept cleanly separated:

1. js/theory.js — music theory, instrument-agnostic
   Anchors everything to MIDI note numbers (C4 = 60). Pitch classes 0–11.
   Exports:
   - NOTE_NAMES_SHARP / NOTE_NAMES_FLAT — 12-element arrays
   - pitchClass(midi), octave(midi), pcToName(pc, useFlats), nameToPc(name),
     midiToName(midi), nameToMidi(name, octave)
   - INTERVALS — named semitone constants
   - SCALES — 15 scale definitions (intervals from root); SCALE_LABELS
     Includes: major, natural/harmonic/melodic minor, pentatonic major/minor,
     blues, dorian, phrygian, lydian, mixolydian, locrian, whole tone,
     diminished (W-H and H-W)
   - CHORDS — 20 chord definitions (intervals + degree labels); CHORD_LABELS
     Includes: major, minor, dim, aug, sus2, sus4, dom7, maj7, min7,
     minMaj7, dim7, halfDim7, aug7, dom9, maj9, min9, add9, dom6, min6, power
   - getScalePitchClasses(rootPc, scaleName) → number[]
   - scaleDegree(pc, rootPc, scaleName) → 1-indexed degree or null
   - getChordPitchClasses(rootPc, chordName) → { pitchClasses, degrees }
   - getChordInversion(rootPc, chordName, inversion) → { pitchClasses,
     degrees, bassNote } — rotates pitch class array so chosen degree is bass
   - relativeMinor(majorRootPc), relativeMajor(minorRootPc), parallelMinor()
   - diatonicChords(rootPc, scaleName) → array of { degree, romanNumeral,
     rootPc, chordName } for all 7 scale degrees with correct case/suffix
     (e.g. "ii", "vii°")

2. js/fretboard.js — guitar fretboard layout, depends on theory.js
   Strings numbered 1–6 (1 = lowest). Fret 0 = open string.
   Exports:
   - parseTuning(spec) — accepts 6 MIDI numbers or note-name strings
   - TUNINGS — 10 named tunings (standard, drop D, open G/D/E/A, DADGAD,
     half/full step down, drop C); TUNING_LABELS
   - buildFretboardMap(tuningSpec, maxFret) → number[][] — MIDI note at
     every [stringIndex][fret] position
   - findPositions(targetPcs, tuningSpec, maxFret) → array of
     { string, fret, midi, pc, degreeIndex } for every matching position
   - findBoxVoicing(targetPcs, tuningSpec, windowStart, windowSize) → one
     position per string (or null) within a fret window
   - filterByFretRange(positions, minFret, maxFret) → filtered array

3. js/renderer.js — SVG fretboard drawing, depends on browser DOM only
   Exports:
   - renderFretboard(container, positions, degreeLabels, opts) — draws the
     fretboard as inline SVG: wood-coloured body, nut, strings (varying
     thickness), fret lines, inlay dots at 3/5/7/9/12 (double at 12), fret
     number labels, coloured dots with degree labels at every note position.
     Open-string notes appear left of the nut. Uses viewBox + CSS width:100%
     so it scales responsively.
   - renderLegend(container, degreeLabels, noteNames, opts) — coloured
     swatches with degree + note name labels.
   - Dots are coloured by degree index (root = red, 3rd = blue, 5th = green,
     etc.). All colours and layout constants are overridable via opts.

4. js/app.js — application shell, wires the above three together
   Manages UI state: mode (chord/scale), root note, chord/scale type, tuning,
   max fret (slider 5–24), flat/sharp toggle. Populates all dropdowns from
   the data in theory.js and fretboard.js. Re-renders on every change.
   Uses DOMContentLoaded so it is safe to load at end of <body>.


Build system
------------
Source files are ES modules (import/export). Because browsers block cross-file
ES module loads from file:// URLs (CORS/null-origin restriction), the app is
bundled before use.

Makefile targets:
  make bundle      — bun build js/app.js → bundle.js (single plain script,
                     no import/export, ~16 KB). Required before opening
                     index.html.
  make standalone  — runs bundle, then scripts/build-standalone.py, which
                     uses regex substitution to inline css/styles.css and
                     bundle.js into index.html, producing
                     guitar-standalone.html (~22 KB). Zero external
                     references. The primary distribution target.
  make serve       — runs bundle then python3 -m http.server 5050
  make test        — bun test (runs against ES module sources directly)
  make clean       — removes bundle.js and guitar-standalone.html

bundle.js and guitar-standalone.html are both committed to the repo so the
app is immediately usable after cloning without running make.


Tests
-----
tests/theory.test.js and tests/fretboard.test.js use bun:test (Jest-like).
179 tests total. Import directly from the ES module source files.
Coverage includes: all note/MIDI helpers, all 15 scales, all 20 chords,
inversions for triads and 7th chords, relative/parallel key relationships,
diatonic chord analysis for major and natural minor, parseTuning, all 10
named tunings, buildFretboardMap, findPositions (including open strings,
cross-octave pitch classes, tuning variants, boundary conditions),
findBoxVoicing, filterByFretRange.


Design constraints
------------------
- No runtime external dependencies (no CDN, no fonts, no network requests)
- All new features should work in the standalone single-file build
- Exceptions (e.g. audio sample fetching) require explicit justification
